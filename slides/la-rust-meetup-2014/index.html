Web Browsers and Rust

A Brief History

  • Around 2011 Mozilla decided it wanted to experiment with a next-generation parallel browser engine. (This project now exists: it's called Servo.)
  • The first question: What language?
  • Everything other than C++ would be a performance loss…
  • …But doing extremely fine-grained parallelism in C++ would be like "juggling chainsaws" (Brendan Eich)

Web browser demands

  • Web browsers are in an extremely constrained space.
  • Modern browsers have the basically the same extreme performance requirements as games do, but are also security-critical.
  • Consumers choose browsers based on speed, but the browser is the safeguard of all your personal information.
  • Little room for compromise!

Enter Rust

  • Development really started in 2010-2011. (Before that, it was basically just some sketches on paper.)
  • Achieved self-hosting in 2011.
  • Massive language changes from 2011 to present.
    • We originally didn't think we could get the language as low-level as it is today while remaining safe.
    • 2011 Rust: somewhat like a compiled Java.
    • 2014 Rust: hosts multiple proof-of-concept OS kernels.
  • On track for release or RC of 1.0 by the end of 2014.

Whirlwind Tour of Rust Features

Performance

  • Rust strives to be as fast as modern, idiomatic C++.
  • No garbage collection, direct control over memory management, SIMD, inline asm…
  • We use LLVM in the same way clang does.
  • There's always work to be done, but we have promising benchmark numbers (e.g. Servo, serde).

Productivity

  • To name a few modern productivity features in Rust:
    • A Python-like, headerless module system
    • Closures
    • Pattern matching
    • Generics with concepts (no more confusing template errors)
    • Interfaces
    • Lisp/Clojure-like macros for compile-time code generation

Safety

  • Memory-safe and data-race free.
  • If safe Rust segfaults, it's our bug, not yours.
  • The safety features have a learning curve, but in my experience they speed you up overall, because you spend less time in front of the debugger.
  • Safety and productivity go hand-in-hand: you spend more time thinking about your problem and less time chasing down segfaults.
  • Not to mention the security benefits!

Community

  • The Rust community is growing!
  • Lots of people interested in performance (e.g. game devs) and safety (e.g. functional programmers).
  • Absurdly talented people from all over the world contributing to the core language.
    • Seriously, I file bugs and then I wake up to pull requests with fixes the next day.
  • An active set of IRC channels on irc.mozilla.org: #rust, #rust-gamedev, #rust-osdev, #rust-webdev, #rust-internals…
